# contract-first — 불신을 기본값으로

외부 입력도, 모델 출력도, 미래의 자기 자신도 일단 의심한다. 신뢰는 검증으로 번다.
이 스킬은 기계가 강제하지 못하는 규율층이다 — finish-the-work(철학 5, "테스트까지가 완료")와 함께 작동한다.

## 언제

새 코드를 쓰거나 고칠 때, 특히 신뢰 경계를 넘는 데이터(사용자 입력, API·모델 응답, 파일, 환경변수, 시계)를 다룰 때.

## 절차

1. **경계에서 불신한다** — 신뢰 경계(외부→내부)를 식별하고, 그 지점의 입력을 "적대적일 수 있다"고 가정한다. "이 값이 거짓말이면?"을 먼저 묻는다.

2. **의미값을 재조립한다** — 의미가 중요한 값은 외부/모델이 준 것을 그대로 쓰지 말고 권위 소스로 재구성한다.
   - 모델이 준 날짜를 버리고 시스템 시계로. ID·해시·타임스탬프·순번은 코드가 생성. 모델 출력에서는 *판단*만 취하고 *사실*은 엔진이 만든다.

3. **범위를 clamp/매핑한다** — 수치·열거는 유효 범위로 강제한다.
   - 수치: `max(lo, min(hi, x))`. 열거: allowlist 매핑, 미지값은 기본값 또는 명시적 거부.

4. **경계값 4종으로 테스트를 두른다** — 각 입력 계약마다 최소:
   - **정상**(happy) · **매핑**(경계·변환·범위 끝) · **None**(누락·빈값·null) · **변조**(악의·형식오류·범위초과)

5. **계약을 글로 남긴다** — 함수·모듈의 입력 가정(타입·범위·null 허용·실패 동작)을 주석/독스트링에 명시한다. 중요한 설계 선택이면 `horos decide D<n> <why> <cost> <escape>` 로 외화한다(→ 철학 2, decision-guard 가 `# D<n>` 주석 참조를 검사).

6. **검증 가능한 주장을 떨군다 (감사층)** — 신뢰 경계를 다루는 변경을 끝냈으면, 무엇을 했고 무엇이 참인지 `horos claim` 으로 외화한다. 엔진(claim-guard)이 *의미 진리가 아니라* 주장의 형식·바인딩 신선도·내부 불변식·교차참조를 결정론으로 판정한다. 예: `horos claim completion "입력 검증+경계값4종 추가" "parse.py,tests/test_parse.py" "" "tests_added"`. (LLM 은 무엇을 주장할지만 정하고, 판정은 엔진이 한다 — 자기검토를 자기에게 맡기지 않는다.)

## 완료 전 체크리스트

- [ ] 외부·모델 입력에 검증/정규화가 있는가
- [ ] 수치·열거가 clamp/매핑되는가 (미지값 처리 포함)
- [ ] 의미값(날짜·ID 등)을 권위 소스로 재조립했는가
- [ ] 경계값 4종(정상/매핑/None/변조) 테스트가 있는가
- [ ] 입력 계약이 글(주석/독스트링)로 적혔는가
- [ ] 변경에 대한 검증 가능한 claim 을 `horos claim` 으로 떨궜는가 (테스트 추가 시 `tests_added`)

## 안티패턴

- 모델/외부가 준 값을 검증 없이 신뢰 경계 안으로 들임
- happy path 테스트만 작성하고 None·변조를 생략
- 범위 없는 수치를 그대로 사용(clamp 누락)
- "이번엔 괜찮겠지" — 미래의 자기 자신도 의심 대상이다
